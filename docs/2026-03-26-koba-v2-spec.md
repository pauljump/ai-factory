# Koba v2 — Design Specification

**Date:** 2026-03-26
**Status:** Designed — ready for implementation

---

## What Koba Is

Koba is a CLI tool that turns a collection of projects into a compounding production environment. You install it globally, point it at your existing projects, and it creates a factory workspace — shared code extracted, domain knowledge harvested, hooks wired, everything managed.

**The thesis:** AI alone produces prototypes. AI + a compounding system produces production-grade software at scale.

**The mechanism:** Koba scans your actual source code, finds duplicated patterns, identifies shared infrastructure, harvests domain knowledge, and organizes everything into a workspace that gets smarter with every project you build.

---

## Core Principle: Claude Code Does the Heavy Lifting

Koba is not a standalone static analysis tool. It is a **system that orchestrates Claude Code sessions** to do the intelligent work. The CLI provides structure, the knowledge store provides memory, the hooks provide enforcement — but the actual scanning, analysis, pattern detection, knowledge harvesting, and code extraction are done by Claude Code.

This means:
- The "deep scanner" is a Claude Code session that reads source code and produces structured analysis, not an AST parser
- The "conversion plan" is Claude Code's judgment informed by scan results, not a deterministic algorithm
- "Pattern fingerprinting" is Claude Code reading multiple projects and recognizing similar code, not hashing
- "Knowledge harvesting" is Claude Code reading code and extracting non-obvious facts, not keyword matching
- Code extraction is Claude Code refactoring real code, not a mechanical copy-paste tool

The lightweight mechanical parts (file discovery, git history, dependency graph from package.json, directory structure creation) are TypeScript. Everything that requires judgment is Claude Code.

## Architecture

### Two Layers

**The tool (updatable):** A globally installed npm package. Contains the CLI, knowledge store, hook scripts, templates, and lightweight mechanical utilities. The CLI orchestrates Claude Code sessions for intelligent work. Updated via `npm update -g koba`.

**The workspace (user's content):** Created by `koba init`, populated by `koba convert`. Contains the user's projects, shared packages, knowledge base, config. Persists across tool updates. Backed up, committed, shared like any repo.

### Workspace Structure

```
my-factory/
  packages/             — shared code (adopted + extracted by koba)
  projects/             — all projects (active work)
  projects/_archive/    — dormant/killed projects
  knowledge/            — structured domain expertise (markdown files)
  data/                 — factory database: FTS5 index, analytics (gitignored)
  scorecards/           — per-project metrics
  koba.json             — factory config
  CLAUDE.md             — factory operating system (generated + customizable)
  .claude/
    settings.json       — hooks (wired automatically by koba)
  package.json          — pnpm workspace root
  pnpm-workspace.yaml   — workspace config
```

### Tool Structure (npm package)

```
koba/
  bin/
    koba.ts             — CLI entry point
  src/
    commands/
      init.ts           — create factory workspace
      convert.ts        — ingest existing projects
      scan.ts           — analyze workspace
      new.ts            — create new project
      status.ts         — factory dashboard
      knowledge.ts      — manage knowledge (add, search, curate)
    engine/
      deep-scanner.ts   — source-level code analysis
      pattern-match.ts  — near-duplicate detection via code fingerprinting
      extractor.ts      — shared code extraction
      knowledge-store.ts — FTS5 knowledge engine

      project-claude.hbs — per-project CLAUDE.md template

        session-start.sh
        stop.sh
  tests/
    ...
```

## CLI Commands

### `koba init [name]`

Creates a new factory workspace.

**Input:** Optional directory name (default: current directory).

**Process:**
1. Create directory structure (packages/, projects/, knowledge/, data/, scorecards/)
2. Generate default koba.json
3. Generate CLAUDE.md from template
4. Initialize pnpm workspace (package.json + pnpm-workspace.yaml)
5. Initialize knowledge store (empty SQLite + FTS5 index)
6. Wire hooks in .claude/settings.json
7. Git init

**Output:** Working factory workspace ready for `koba convert` or `koba new`.

### `koba convert <source> [--dry-run]`

Ingests an existing monorepo or collection of projects into the factory.

**Input:** Path to source directory. Can be a monorepo, a single project, or a directory containing multiple project directories.

**Process:**

**Phase 1 — Deep Scan:**
For each project directory found:
- Language/framework detection
- Full dependency graph (npm packages + relative imports between files)
- Export analysis (what the project exposes)
- Pattern fingerprinting: hash code blocks for common patterns (auth, DB setup, API clients, error handling, middleware, deploy config) to detect near-duplicates across projects
- Domain knowledge extraction: scan comments, variable names, READMEs, CLAUDE.md files, git commit messages for hard-won facts
- Git history: commits, session estimation, time to deploy, activity recency

Across all projects:
- Duplicate detection (which projects share near-identical code blocks)
- Dependency overlap (what npm packages appear in N+ projects)
- Existing shared packages: if source has a packages/ directory, validate each package is actually imported by 2+ projects. Flag dead ones.
- Missing extractions: code patterns in 3+ projects not yet shared
- Domain clustering: group projects by detected domain for knowledge organization

**Phase 2 — Conversion Plan:**
AI-assisted analysis of scan results. Produces a structured plan:
- ADOPT: existing shared packages to keep (with consumer count)
- EXTRACT: new shared code candidates (with source projects and best implementation)
- ACTIVE: projects to import into projects/
- ARCHIVE: dormant projects (no commits in 90+ days, no deploy config)
- MERGE/KILL: redundant projects (overlap with active ones)
- KNOWLEDGE: harvested entries by domain with count

With `--dry-run`: displays the plan and exits.
Without: displays the plan and prompts for approval section by section.

**Phase 3 — Execute:**
After approval:
1. Copy adopted packages to packages/
2. Copy active projects to projects/
3. Copy archived projects to projects/_archive/
4. Execute approved extractions (create new packages, update imports across consuming projects)
5. Build knowledge base from all harvested sources
6. Generate koba.json from detected stack, conventions, patterns
7. Generate CLAUDE.md from factory state
8. Wire hooks in .claude/settings.json
9. Generate pnpm-workspace.yaml listing all packages and projects
10. Run `pnpm install`
11. Run available tests across projects to verify nothing broke
12. Output conversion report with summary stats

### `koba scan`

Re-analyzes the workspace. Run after building projects or periodically.

**Process:**
1. Run deep scanner on all active projects
2. Compare against last scan: new patterns, new extraction candidates, stale knowledge
3. Regenerate CLAUDE.md (preserving user-customized sections)
4. Update koba.json if new patterns detected
5. Output findings

**Output:** List of findings — extraction candidates, stale knowledge, health issues.

### `koba new <name>`

Creates a new project in the factory.

**Process:**
1. Interactive: ask project type (web, api, ios, data-pipeline, other)
2. Scaffold from template appropriate to type
3. Wire relevant shared packages
4. Generate project CLAUDE.md with relevant knowledge pre-loaded
5. Add to pnpm workspace
6. Run pnpm install

### `koba status`

Dashboard showing factory health.

**Output:**
- Project count (active/archived)
- Package count with avg consumer count
- Knowledge entry count by domain
- Stale knowledge warnings
- Extraction candidates pending
- PDLC phase and metrics (if tracking)

### `koba knowledge add`

Interactive knowledge entry creation.

**Process:**
1. Prompt for domain, body, tags, source project, confidence
2. Write markdown file to knowledge/
3. Update FTS5 index

### `koba knowledge search <query>`

Search knowledge base from CLI.

**Output:** Ranked results with domain, confidence, source project, body preview.

## The Deep Scanner

The current scanner reads package.json. The deep scanner uses Claude Code to read and understand actual source code. This is the intelligence layer.

### Mechanical Layer (TypeScript — fast, deterministic)

The CLI handles what doesn't require judgment:
- File discovery: list all projects, source files, config files
- Dependency graph: parse package.json across all projects
- Git history: commit dates, session estimation, deploy detection
- Activity classification: active (commit in 30 days), dormant (30-90), dead (90+)
- Directory structure: detect framework by file patterns (project.yml = iOS, etc.)

This produces structured JSON that feeds into the Claude Code analysis.

### Intelligence Layer (Claude Code — judgment, pattern recognition)

Claude Code sessions do the work that requires understanding:
- **Reading source code** across projects to find similar patterns (auth, DB setup, API clients, error handling, middleware)
- **Evaluating** which implementation of a shared pattern is "best" (cleanest, most tested, most maintainable)
- **Harvesting knowledge** from code comments, READMEs, CLAUDE.md files — extracting non-obvious facts that a keyword search would miss
- **Proposing extractions** with reasoning about what to share and what to keep project-specific
- **Domain clustering** — understanding that stuywatch and vayo are both NYC real estate projects, not just "they both use SQLite"

The CLI orchestrates this by providing Claude Code with the mechanical scan results plus the relevant source files, and collecting the structured output.

### What It Produces

```typescript
interface ScanResult {
  projects: ProjectScan[]                      // per-project analysis
  patterns: PatternMatch[]                     // similar code across projects (Claude-identified)
  extractionCandidates: ExtractionCandidate[]  // recommended new shared packages
  knowledgeEntries: KnowledgeEntry[]           // harvested domain knowledge
  packageHealth: PackageHealth[]               // existing packages: consumers, dead code
  domainClusters: DomainCluster[]              // projects grouped by domain
}
```

## Knowledge Engine

Same core as Phase 0 (markdown files + SQLite FTS5) with improvements:

**Storage:** Structured markdown files in knowledge/<domain>/<id>.md with YAML frontmatter. Source of truth is the files (git-trackable, human-readable). SQLite FTS5 index is derived and rebuilt from files.

**Injection:** SessionStart hook detects project context (directory, dependencies, framework), queries the index, injects the top 15 most relevant entries. Priority: same-source-project > tag-match > full-text-search.

**Capture:** Stop hook prompts (via the AI conversation, not the hook itself) for new knowledge entries. User approves or rejects. Approved entries written to knowledge/ and indexed.

**Harvesting:** During convert and scan, the scanner extracts knowledge from code comments, READMEs, CLAUDE.md files, and git messages. These are proposed as entries for user approval (not auto-committed).

**Curation:** `koba scan` flags entries not accessed in 90 days, detects near-duplicates, tracks precision (entries injected vs. entries marked useful).

## CLAUDE.md Generation

Koba generates the factory's CLAUDE.md from its actual state. Two types of sections:

**Auto-generated (regenerated on `koba scan`):**
- Stack (from koba.json + detected patterns)
- Active projects table (from projects/ + git activity)
- Shared packages table (from packages/ + consumer count)
- Knowledge domains summary (from knowledge/)
- Factory rules (from koba.json)

**User-written (preserved across regeneration):**
- How We Work (collaboration style, roles, principles)
- Who You're Working With (user identity)

The template uses markers to distinguish:
```markdown
<!-- koba:auto-start:active-projects -->
| Project | Status | Last Active | Packages Used |
...
<!-- koba:auto-end:active-projects -->

<!-- koba:user-start:how-we-work -->
[User's content here — never overwritten]
<!-- koba:user-end:how-we-work -->
```

## Hook System

Hooks are shell scripts generated by `koba init` into the workspace's `.claude/settings.json`. They call back into the globally installed koba CLI.

**SessionStart:**
```bash
koba _hook session-start --cwd "$CWD"
```
Detects project context, queries knowledge, outputs relevant entries to stdout. Claude Code adds stdout to session context.

**Stop:**
```bash
koba _hook stop --cwd "$CWD"
```
Logs session metrics to analytics.

The `_hook` subcommand is internal (prefixed with underscore). It reads the workspace's data/factory.db for knowledge and analytics.

This is cleaner than shell scripts calling tsx — it's just the koba CLI being invoked by the hook.

## Sub-Project Build Order

### Sub-project 1: CLI skeleton + init
Create the npm package structure with bin entry, command routing, and the `init` command. After this: `koba init` works and creates a factory workspace.

### Sub-project 2: Deep scanner
Replace the lightweight scanner with source-level analysis. Pattern fingerprinting, knowledge harvesting, git history. After this: `koba scan` produces rich analysis.

### Sub-project 3: Convert command
The full conversion engine. Uses the deep scanner, produces a plan, executes after approval. After this: you can convert your monorepo.

### Sub-project 4: Knowledge engine integration
Move knowledge store into the CLI. `koba knowledge add/search`. Hook scripts that call `koba _hook`. After this: hooks work from the global install.

### Sub-project 5: Project lifecycle
`koba new` with templates, scaffolding. Scorecard generation. After this: full factory loop for new projects.

### Sub-project 6: CLAUDE.md generation
Auto-generate from factory state. Regenerate on scan. Preserve user sections. After this: the factory OS maintains itself.

### Sub-project 7: Status + polish
Dashboard, health checks, error handling, edge cases, README that matches reality.

## What Gets Rebuilt vs. Reused

**Reused from Phase 0 (works, just needs restructuring):**
- Knowledge store (FTS5 search, CRUD) → moves to src/engine/knowledge-store.ts
- Knowledge entry parser/writer → moves to src/engine/
- Analytics/session tracking → moves to src/engine/analytics.ts
- FTS5 search lib → stays in src/lib/search.ts
- Analytics lib → stays in src/lib/analytics.ts
- Baseline extraction (session estimation, category detection) → moves to src/engine/baseline.ts
- All existing tests (adapted for new paths)

**Rebuilt (current version is insufficient):**
- Scanner → replaced by deep scanner (mechanical layer + Claude Code intelligence)
- CLI → new (currently no CLI, just npm scripts)
- Hooks → simplified (call `koba _hook` instead of tsx scripts)
- Bootstrap → replaced by convert command

**New:**
- CLI framework (bin entry, command routing, argument parsing)
- Claude Code orchestration layer (structured prompts for scan/analyze/extract tasks)
- Knowledge harvesting via Claude Code (reads source, extracts non-obvious facts)
- Conversion plan generation and execution
- CLAUDE.md generation with auto/user section markers
- koba.json config system
- `koba new` scaffolding with knowledge pre-loading
- `koba status` dashboard
